📜 起源：大包大揽 vs 小步快跑

🎨 设计迭代的两种模式

方式一（大包大揽）：
"帮我做个页面，要好看，要有动效，要有交互，要适配移动端，要……"
设计师奋战三天，给你一个巨大的 Figma 文件。
你看完说："颜色不太对，改一下。"
设计师崩溃了——这个改动牵扯了整个设计系统的颜色变量。

方式二（小步快跑）：
"先帮我做个基础的页面结构。"
"颜色改成品牌蓝 #0ea5e9。"
"间距再大一点。"
"左边加个侧边栏。"
每一步都很小，每一轮都看效果，方向不对随时调。

Codex 推荐的 UI 迭代方式就是第二种——
用小而具体的 prompt 迭代，每轮只改一个维度。

📖 截图做原型

CLI 工作流（图 + prompt）

步骤：

  1. 把截图存到本地（例如 ./specs/ui.png）
  2. 运行 Codex：codex
  3. 把图片文件拖进终端，附到 prompt 上
  4. 给出约束和结构：

Prompt：

照这张图做一个新的 dashboard。
约束：用 react、vite 和 tailwind。代码写 typescript。间距、字体和布局尽量贴近原图。
产出：一个渲染这个界面的新路由/页面，需要用到的小组件，README.md 写明本地怎么跑起来。

⚠️ 两点注意：
  1. 图片给的是视觉要求，实现层面的约束（框架、路由、组件风格）还要你自己写清
  2. 图里看不出来的行为要用文字补上——悬停状态、校验规则、键盘交互

验证方式： 让 Codex 启动 dev server，告诉你在本地哪个 URL 能看到原型。

📖 迭代 UI：设计→微调→刷新→再微调

CLI 工作流

  1. 启动 Codex
  2. 另开一个终端窗口启动 dev server：npm run dev
  3. 让 Codex 提改动：

第一轮： 给落地页提两三个样式改进方案。

第二轮（挑一个方向，小步迭代）： 就按方案二来。只改头部：字体排印做得更有杂志感。留白加大。保证移动端依然好看。

第三轮： 下一轮：降低视觉噪音。布局保持，颜色做减法，多余的边框都去掉。

关键原则

做不要做
每轮只改一个维度（只改颜色、只改间距、只改字体）一轮改十几个东西
在浏览器里实时看效果等所有改动全写完了再看
喜欢的就提交，不喜欢的就回滚保留所有改动不舍得删
你回滚或手改了某处，告诉 Codex让 Codex 覆盖你的手工修改
