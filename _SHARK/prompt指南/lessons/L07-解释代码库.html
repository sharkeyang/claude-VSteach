📜 起源：为什么"帮我看看这段代码"不管用？

🏗️ 接手一个"祖传代码"

你刚加入一个新项目，同事丢给你一个模块：
"这是我们的鉴权模块，你看看。"

你打开文件，看到 500 行代码，5 个函数，十几个 import……
从哪看起？

如果只是说"帮我看看这段代码"，AI 会给你一个泛泛的概括——
这不是你想要的。

你想要的是：
这个请求从哪里进来？经过了哪些模块？每个模块干了什么？改的时候要注意什么坑？

给 Codex 的 Prompt 越具体，它的分析就越有价值。
关键是以问题驱动——你想知道什么，就明确问什么。

📖 两种工作流

方式一：IDE 插件（本地探索最快）

步骤：

  1. 打开最相关的文件
  2. 选中你关心的代码（可选，但推荐）
  3. 给 Codex 下 prompt：

Prompt：

解释请求是怎么流经这段选中代码的。包含：
- 涉及的每个模块，职责的简短总结
- 哪些数据在哪里被校验
- 改这段代码时要留神的一两个「坑」

验证方式： 让它给一份你能核对的图或清单——"把请求链路总结成一个带编号的步骤列表，然后列出涉及的文件"。

方式二：CLI（适合完整记录和 shell 命令）

步骤：

  1. 启动交互会话：codex
  2. 附上文件（可选），然后下 prompt：

Prompt：

我需要理解这个服务用的协议。读一下 @foo.ts @schema.ts 并解释 schema 和请求/响应链路。重点讲必填字段和可选字段的区别，以及向后兼容的规则。

💡 上下文说明：

  • IDE： 自动把你打开的文件带进上下文。选中代码可进一步缩小范围。
  • CLI： 用 @ 插入工作区文件路径，或用 /mention 附上指定文件。

Codex Prompt 的通用结构

要素说明示例
🎯 行为你要 Codex 做什么解释请求链路 / 分析协议
📚 相关代码指向文件或代码段@foo.ts @schema.ts
