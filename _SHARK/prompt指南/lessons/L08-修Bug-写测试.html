📜 起源：为什么"帮我修个 Bug"不够？

🏥 看医生 vs 修 Bug

你去看医生：
"我不舒服。" → 医生没法诊断。
"我发烧 38.5℃两天了，咳嗽有黄痰，周一吃了冰的之后开始的。" → 医生能开药了。

修 Bug 也一样——
"有个 Bug，保存按钮点了没反应。"
→ Codex 没法定位，只能猜。

"Bug：在设置页点「保存」，有时显示「已保存」但改动没有真正落盘。
复现步骤：1）npm run dev 2）打开 /settings 3）切换开关 4）点保存 5）刷新页面：开关弹回去了。
约束：不要改 API 的形状。修复保持最小化，可行的话加回归测试。"
→ Codex 能精确复现、定位、修复、验证。

复现步骤 + 约束 + 验证方式，是修 Bug Prompt 的三件套。

📖 修 Bug Prompt 模板

完整 Prompt 结构

修 Bug Prompt = Bug 描述 + 复现步骤 + 约束 + 验证方式

完整示例：

Bug：在设置页点「保存」，有时显示「已保存」，但改动没有真正落盘。

Repro：
1）启动应用：npm run dev
2）打开 /settings
3）切换「启用提醒」开关
4）点击 Save
5）刷新页面：开关弹回去了

Constraints：
- 不要改 API 的形状
- 修复保持最小化，可行的话加一个回归测试

先在本地复现这个 bug，然后提出补丁并跑检查。

为什么这样写？

你在 Prompt 中给的Codex 自动提供的
复现步骤（重要性远高于高层描述）命令输出、找到的调用点
约束条件（不改 API、最小修复）触发的堆栈信息

验证方式

修完之后：

修完之后，跑 lint 加上最小相关的测试集。报告用了哪些命令、结果如何。

Codex 应该重跑一遍复现步骤——如果复现不了，才算真修好了。

✏️ 写测试 Prompt 模板

方式一：IDE 插件（基于选中代码）

  1. 打开包含那个函数的文件
  2. 选中定义这个函数的那几行
  3. 在命令面板选「Add to Codex Thread」
  4. 下 prompt：

Prompt： 给这个函数写一个单元测试，遵循其他测试里的惯例。

方式二：CLI（在 prompt 里写路径和函数名）

Prompt： 给 @transform.ts 里的 invert_list 函数加一个测试。覆盖正常路径，加上边界情况。
