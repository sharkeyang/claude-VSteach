📋 终检清单

每次写完 Prompt 后，快速过一遍：

  1. ☐ 从结果说起，还是从步骤说起？ → 改成结果
  2. ☐ 上下文够不够？ → 只加相关的，说明取什么
  3. ☐ 受众和格式写了吗？ → 加一句"给谁看、多长、什么格式"
  4. ☐ 最关键的边界设了吗？ → 改错就废的东西，保护起来
  5. ☐ 要终检吗？ → 重要工作让 AI 检查遗漏
  6. ☐ 选对模式了吗？ → Chat 轻量 / Work 复杂 / Codex 代码

📐 常用模板

📝 日常 Chat 模板

为 [受众] [做什么]。 要求：[语气/风格]。 长度：[字数/页数]。 约束：[关键限制]。

示例：

为不懂技术的人解释区块链是什么。 用一个生活化的类比。控制在 300 字以内。 不要用技术术语。

📊 Work 模板（复杂任务）

用 [附件/来源]，做一份 [产出物]。 受众是 [谁]。 [决策/关键发现] 放最前面。 [数据] 和 [分析] 要区分开。 每个数字注明来源。 收尾前检查 [一致性]。 不要发送或发布，给我审完再说。

🐛 修 Bug 模板

Bug：[现象描述] Repro： 1）[第一步] 2）[第二步] 3）[第三步] Constraints： - [不改的约束] - [最小化要求] 先在本地重现，定位根因，提出修复，跑测试验证。

🖼️ 截图做原型模板

照这张图做一个 [页面名称]。 约束：用 [框架] [工具] [语言]。 要求：间距、字体、布局尽量贴近原图。 图里看不到的行为： - [行为1，如 hover 状态] - [行为2，如校验规则] 产出：[可运行页面 / 组件]。

🔍 解释代码库模板

解释 [请求/数据] 是怎么流经 [文件/模块] 的。 包含： - 每个模块的职责 - 哪里做了校验 - 改的时候要注意的坑

👀 Code Review 模板

/review 重点看 [边界情况/安全问题/性能]

⚠️ 常见坑

#坑正确做法
1从步骤说起"先打开……再……"从结果说起"做一张……"
2上下文太少，AI 猜偏只加相关来源，说明取什么
3上下文太多，稀释注意力只加真正相关的，删掉无关的
4没设边界，AI 改了不该改的加一两条关键边界
5没做终检，数据/结论不一致让 AI 收尾时自查一遍
6第一条 prompt 想完美，半天写不出来先写个大概，用追问迭代
7修 Bug 只给症状不给复现步骤给精确的复现步骤 + 约束
8给图片不说明关键区域指出图里关键区域和交互行为
9不看 AI 的产出直接发出终检 + 自己再过一遍
10所有任务都用 Chat复杂任务用 Work，代码用 Codex

🏆 黄金法则

  1. 从结果开始 — 先说什么，再说怎么做
  2. 只加有用的上下文 — 多了反而干扰
  3. 边界要少而精 — 只保护最关键的约束
  4. 说清怎么用结果 — 受众、格式、长度
  5. 第一条不用完美 — 追问是正常流程
  6. 选对工具 — Chat 聊日常，Work 做复杂活，Codex 写代码
  7. 一定要验证 — 终检 + 自审，不信任 AI 的最后一公里
