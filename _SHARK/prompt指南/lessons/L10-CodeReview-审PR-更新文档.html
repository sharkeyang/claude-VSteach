📜 起源：为什么需要"第二双眼睛"？

👀 你写了一下午代码

你埋头写了四个小时的代码，终于跑通了。
提交前你自己看了三遍——"没问题了。"
把 PR 发出去，同事五分钟就发现了一个边界情况没处理。

不是你不认真——是你太熟悉自己的代码了。
熟视无睹是人性，Code Review 就是对抗这个的机制。

AI 做 Code Review 的优势：
- 它不累——看第 1000 行和看第 1 行一样专注
- 它不会"审美疲劳"——不会因为代码风格跟你一样就放过潜在问题
- 你可以指定它重点看什么——安全、边界情况、性能……

但要注意：AI 的 Review 是辅助，不是替代。 最终把关的还是你自己。

📖 三种 Codex 质量场景

场景一：本地 Code Review

适用场景： 提交或建 PR 之前，想找双眼睛帮你把把关。

CLI 工作流：

  1. 启动 Codex：codex
  2. 跑审查命令：/review
  3. 可选：给出自定义关注点

指定关注点： /review 重点看边界情况和安全问题

验证方式： 按审查反馈修完，再跑一遍 /review，确认问题都解决了。

场景二：审一个 GitHub PR

适用场景： 不想把分支拉到本地，也能拿到审查反馈。

前置条件： 先在仓库上启用 Codex Code review 功能。

GitHub 工作流（靠评论驱动）：

  1. 在 GitHub 上打开那个 PR
  2. 留一条评论 @ Codex，写明关注点：@codex review
  3. 可选：给更明确的指令：@codex review 重点审安全漏洞和安全隐患

场景三：更新文档

适用场景： 需要一次准确、清晰的文档修改。

Prompt： 更新「高级功能」文档，补充鉴权排障指引。检查所有链接都有效。

要点： 用 @ 指定要改的文档文件，写明范围和验证要求。

验证方式： 读一遍渲染后的页面。

速查表

场景用什么关键 Prompt / 命令验证方式

🌩️ 补充：把重构交给云端

当重构任务太大，不想阻塞本地工作时，可以在本地规划好方案，交给云端执行。

本地规划（IDE）：

$plan 我们要重构鉴权子系统，目标：拆分职责（token 解析、会话加载、权限各归各）。减少循环引用。提升可测试性。约束：用户可见的行为不能变。公开 API 保持稳定。给出分步骤的迁移计划。

云端委派： 审完计划后，切到云环境，输入"实现计划里的里程碑一"。

事后： 审云端的 diff，需要就迭代，最后直接从云端建 PR 或拉到本地测试收尾。
