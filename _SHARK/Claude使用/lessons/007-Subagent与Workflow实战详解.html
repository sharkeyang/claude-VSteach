第 12 课 · Subagent 与 Workflow 实战详解

这是本课程最硬核的一课。Subagent 和 Workflow 是 Claude Code 最强大的高级功能，掌握它们等于从"一个人干活"升级到"指挥一个团队干活"。

第一部分：Subagent（子智能体）

Subagent 的核心概念

Subagent 就是"再开一个 Claude"——它独立运行，有自己的上下文，干完活把结果告诉你。你可以同时开多个，互不干扰。

对比普通对话Subagent
工作方式一个一个来多个同时干
上下文共享同一个对话每个 agent 独立
适合需要你来决策的对话独立任务、批量操作
效率串行，慢并行，快

启动方式

在对话中直接说"帮我启动一个 agent 做 XXX"，或者用 @agent 语法：

启动 Subagent 的三种方式
# 方式一：自然语言（推荐）
帮我启动一个 agent，搜索 _CHRIS/ 目录下所有包含 "暑假" 的文件

# 方式二：指定类型
启动一个 Explore agent，查找所有 .py 文件中没有 type hints 的函数

# 方式三：指定类型 + 隔离模式
启动一个 worktree agent，帮我重构 _CHRIS/__temp/gen_calligraphy.py

实战场景一：并行搜索 🔍

场景：项目中有大量文件，想同时搜索多个关键词
❌ 不用 Subagent：搜完一个关键词，等结果，再搜下一个，串行耗时长
✅ 用 Subagent：
你输入：
帮我启动三个 agent：

agent 1：搜索所有 .py 文件中包含 "TODO" 的行，统计数量并按文件列出

agent 2：搜索所有 .py 文件中包含 "FIXME" 的行，统计数量并按文件列出

agent 3：搜索所有 .py 文件中包含 "HACK" 或 "XXX" 的行，统计数量并按文件列出
→ Agent 1 启动，搜索 TODO...
→ Agent 2 启动，搜索 FIXME...
→ Agent 3 启动，搜索 HACK/XXX...
（三个 agent 同时搜索，总时间 ≈ 搜一个关键词的时间）
结果：三个结果同时返回，汇总成一份完整的代码待办事项清单。

实战场景二：并行改文件 ✏️

场景：有 6 个 Python 脚本，都要在文件头部加同一段版权声明
❌ 不用 Subagent：一个文件一个文件改，每次都要等 Claude 切换上下文
✅ 用 Subagent：
你输入：
把以下文件分成三组，启动三个 agent 同时加版权声明头：

文件列表：zdata.py, QMT_查询.py, AKshare_获取交易日.py, miniQMT连接服务器.py, miniqmt下载数据2.py, 下载周月线_验算用.py

版权声明内容：
# Copyright (c) 2026 SharkWork. All rights reserved.
# 本文件仅供个人学习使用
→ Agent 1 处理 zdata.py + QMT_查询.py
→ Agent 2 处理 AKshare_获取交易日.py + miniQMT连接服务器.py
→ Agent 3 处理 miniqmt下载数据2.py + 下载周月线_验算用.py
（三个 agent 同时改，总时间 ≈ 改两个文件的时间）

实战场景三：独立调研 🔎

场景：你在写代码，同时想查一个技术问题，不想中断
❌ 不用 Subagent：停下当前工作，去查资料，思路被打断
✅ 用 Subagent：
你输入：
启动一个 agent，帮我查一下：
akshare 库最新版本是否支持获取北交所股票的日线数据？
如果支持，给一个示例代码。
→ Agent 去查资料（可能需要联网 MCP）
→ 你继续手头的工作
→ Agent 查完后回来报告结果
效果：不用切换上下文，调研结果自动回来，工作流不中断。

实战场景四：审查 + 修复流水线

场景：刚写完一个功能，想同时审查代码和修复发现的问题
你输入：
启动两个 agent：

agent 1（审查）：对 zdata.py 做代码审查，列出所有潜在问题——包括语法问题、性能问题、安全隐患、代码风格问题。按严重程度排序。

agent 2（修复）：等 agent 1 完成后，根据它找到的问题逐个修复。
→ Agent 1 开始审查 zdata.py
→ Agent 1 完成，列出问题清单
→ Agent 2 启动，开始修复问题
→ 你可以在 agent 工作期间去做其他事

Subagent 的隔离模式详解

模式语法什么时候用
普通@agent 做XXX只是读文件、搜索、分析，不改文件
worktree启动一个 worktree agent，做XXX多个 agent 要同时改文件，担心互相覆盖
remote启动一个 remote agent，做XXX需要远程计算资源或隔离环境
💡 什么时候用 worktree 隔离？ 当多个 agent 要同时修改同一个项目的不同文件时，用 worktree 模式让每个 agent 在独立的分支上工作，改完后自动合并。 如果只是读文件或搜索，用普通模式即可，不需要 worktree。

第二部分：Workflow（多智能体工作流）

Workflow 的核心概念

如果说 Subagent 是"派一个人去干活"，那 Workflow 就是"组织一条流水线"。你可以定义多个阶段，每个阶段多个 agent 并行工作，前一个阶段的输出成为后一个阶段的输入。

对比SubagentWorkflow
关系各自独立有编排、有依赖
流程一个任务多阶段流水线
协作互不通信前一个的输出给后一个
适合单次独立任务多步骤、多角色的复杂流程

实战场景五：代码全面审查流水线 🔍

场景：提交代码前，想从多个维度全面审查，确保质量
设计：3 个 agent 从不同角度并行审查，然后汇总
你输入：
帮我启动一个审查工作流，对当前的代码改动做全面审查：

阶段一（并行审查）：
agent 1（安全角度）：检查有没有 SQL 注入、路径遍历、敏感信息泄露等安全问题
agent 2（性能角度）：检查有没有不必要的循环、重复计算、大文件加载等性能问题
agent 3（可维护性角度）：检查函数是否太长、命名是否清晰、有没有重复代码

阶段二（汇总）：
等三个 agent 都完成后，把结果汇总成一份报告，按严重程度排序，并给出修复建议
→ 三个 agent 同时启动，分别审查
→ Agent 1（安全）完成，找到 2 个问题
→ Agent 2（性能）完成，找到 1 个问题
→ Agent 3（可维护性）完成，找到 3 个问题
→ 汇总 agent 生成综合报告
效果：一次审查覆盖安全 + 性能 + 可维护性三个维度，比逐个审查快 3 倍。

实战场景六：批量迁移脚本 🔄

场景：需要把一批旧脚本迁移到新目录，同时更新 import 路径
设计：先扫描有哪些文件，再分组并行迁移，最后验证
你输入：
启动一个迁移工作流：

阶段一（扫描）：
搜索 .history/ 目录下所有 .py 文件，列出文件名和它们 import 了哪些模块

阶段二（并行迁移）：
把阶段一找到的文件平均分成 3 组，每组启动一个 agent 来做：
1. 把文件复制到 _CHRIS/scripts/ 目录
2. 把文件中的 import 路径从旧路径改为新路径
3. 添加新文件的头部注释

阶段三（验证）：
检查所有迁移后的文件语法是否正确，列出成功/失败的文件
→ 阶段一：扫描到 12 个文件
→ 阶段二：3 个 agent 各迁移 4 个文件，并行进行
→ 阶段三：验证全部通过，12/12 迁移成功

实战场景七：多维度数据质量检查 📊

场景：数据下载完成后，想从多个维度检查数据质量
你输入：
启动一个数据质量检查工作流：

阶段一（并行检查）：
agent 1（完整性）：检查 D:/zdata/dataAK中股/ 目录下每个 CSV 的行数是否合理，有没有空文件
agent 2（时效性）：检查每个 CSV 的最新日期是不是今天，有没有滞后超过 3 天的
agent 3（一致性）：随机抽 10 个 CSV，检查它们的 OHLC 数据是否合理（高≥低、开≥低等）

阶段二（报告）：
汇总三个 agent 的结果，生成一份数据质量报告，列出所有异常
→ 三个 agent 同时检查
→ Agent 1：发现 2 个空文件
→ Agent 2：发现 5 个文件滞后超过 3 天
→ Agent 3：所有抽检数据合理
→ 汇总报告显示具体问题文件和修复建议

实战场景八：Subagent + Workflow 混合使用 🔀

场景：大项目重构，需要同时做多件事，且有些事依赖其他事的结果
设计：前面的 agent 结果出来了，后面的 agent 才能开始
你输入：
帮我做一个项目重构工作流：

阶段一（分析）：
启动一个 agent，分析 zdata.py 中有哪些函数，每个函数做什么，画出函数依赖关系

阶段二（并行设计）：
根据阶段一的结果，启动两个 agent：
agent 1：设计数据下载模块的重构方案
agent 2：设计数据校验模块的重构方案

阶段三（并行实施）：
根据阶段二的两个方案，各启动一个 agent 实施重构

阶段四（验证）：
启动一个 agent 做最终的集成测试，确保重构后功能正常
→ 阶段一：分析完成，发现 zdata.py 有 15 个函数，5 个核心模块
→ 阶段二：两个 agent 并行出方案
→ 阶段三：两个 agent 并行实施
→ 阶段四：集成测试通过

第三部分：选择指南

什么时候用什么？

任务类型推荐方式原因
同时在多个目录搜索关键词🔸 Subagent（并行）独立任务，不需要编排
批量修改多个文件🔸 Subagent（并行）每个文件独立，同时改
边干活边查资料🔸 Subagent（后台）不中断当前工作流
多维度代码审查+汇总🔷 Workflow需要编排，先并行审查再汇总
迁移项目（分析→迁移→验证）🔷 Workflow多阶段，有依赖关系
数据质量检查+报告🔷 Workflow需要多角度检查后汇总
大项目重构🔷 Workflow（多阶段）复杂依赖，需要严格编排

快速决策

🎯 一句话判断：
  • 任务之间没有依赖，只是要同时做 → Subagent
  • 任务有先后顺序，前一个的输出给后一个 → Workflow
  • 任务复杂，既有并行又有依赖 → Workflow 编排 Subagent

总结

功能一句话启动方式典型场景
普通 Subagent派一个人去干活帮我启动一个 agent 做XXX搜索、批量改文件、调研
Worktree Subagent在独立分支上干活启动一个 worktree agent 做XXX多个 agent 同时改文件
Workflow组织一条流水线启动一个工作流：阶段一XXX，阶段二XXX审查→修复→验证、迁移
💡 记住三句话：
  1. 搜索调研用 Subagent — 派出去查，不耽误你干活
  2. 多维度审查用 Workflow — 不同角度同时看，最后汇总
  3. 大项目重构用 Workflow 编排 — 先分析→再设计→再实施→再验证
