第 13 课 · Harness 实施实战

用 Claude Code 命令，从头搭建 Harness 工程体系 · 完整操作实录

前面 12 课学了所有命令和功能，这一课把它们串起来——实操 Harness 五条方法，从零开始给一个项目套上"缰绳"。

我们以一个新项目 D:\@VSwork\demo-project 为例，模拟完整的实施过程。你只需要复制里面的"你说"内容，换成你自己的项目路径即可。

🎯 本课目标： 学完之后你不仅能理解 Harness 是什么，还知道具体每一步该跟 Claude 说什么命令。

阶段一：初始化项目 — /init 生成 CLAUDE.md

📋 步骤 1.1：创建项目并初始化规则 2 分钟
你说：
我已经创建了 demo-project 目录，里面有一个 main.py 文件。
帮我运行 /init，为这个项目生成一份 CLAUDE.md。
→ Claude 扫描目录，分析项目结构
→ 自动生成 CLAUDE.md，包含：项目简介、技术栈、关键文件说明
你说（继续完善）：
在 CLAUDE.md 末尾追加 AI 开发规则章节，包括：

## AI 开发规则

### 工作流程
1. 先方案后编码 — 任何改动前先口头确认方案
2. 重大变更使用 Plan 模式

### 技术边界
- Python 3.11，标准库优先
- 所有路径使用绝对路径 D:\@VSwork\demo-project\

### 代码规范
- type hints + 函数注释
- 不留下注释掉的代码和 print 调试语句

### 质量管理
- 修改后运行 python _test/run_verify.py

### 版本控制
- 每功能一提交，格式：feat:/fix:/docs:
→ CLAUDE.md 已更新，包含 AI 开发规则章节 ✅
✅ 方法一完成：规则文件已创建，以后每次进入项目 Claude 自动读取，知道边界在哪。

阶段二：权限配置 — 创建 settings.json

📋 步骤 2.1：创建 .claude/settings.json 3 分钟
你说：
帮我创建项目的 .claude/settings.json：

1. 权限白名单：cd, python, pip, git, mkdir 等常用操作
2. hooks：Bash 命令前自动 cd 到项目目录
3. allowWarningReminders: false
→ settings.json 已创建 ✅
→ 以后运行 Bash 命令自动进入项目目录，权限操作不再频繁弹窗
✅ 方法三（部分）完成：权限配置到位，工具使用不再被打断。

阶段三：创建验证脚本 — _test/run_verify.py

📋 步骤 3.1：创建验证脚本 5 分钟
你说：
帮我创建 _test/run_verify.py，需要做到：

1. Python 语法检查 — 扫描所有 .py 文件
2. 文件完整性检查 — 检查 CLAUDE.md、settings.json、main.py 是否存在
3. 输出格式用 [PASS]/[FAIL]，不用 emoji（Windows GBK 兼容）
4. 全部通过返回 0，有失败返回 1
→ _test/run_verify.py 已创建 ✅
你说（测试验证）：
运行 python _test/run_verify.py 看看结果
→ [PASS] All checks passed! ✅
✅ 方法四（部分）完成：验证脚本就位，改完代码就能一键检查。

阶段四：Plan 模式开发 — 先方案后编码

📋 步骤 4.1：用 Plan 模式开发新功能 贯穿全程
你说（新需求来了）：
/plan 我想在 main.py 中加一个函数，从 CSV 文件读取数据并计算平均值。出个方案让我看看。
→ Claude 输出方案：
- 修改 main.py，新增 read_csv() 和 calc_average() 两个函数
- 使用 csv 标准库，不引入第三方依赖
- 错误处理：文件不存在、空文件、非数字值
- 不影响现有功能
你说（确认方案）：
方案可以，去掉 calc_average，改成 read_csv 返回所有数据，平均值在外面算。开始编码。
→ Claude 按照确认后的方案编码 ✅
→ 没有多做功能，没有做不需要的 calc_average
你说（验证）：
跑一下验证，确认没问题
→ python _test/run_verify.py → [PASS] ✅
✅ 方法二完成：先方案后编码成为工作习惯，避免方向错了返工。

阶段五：pre-commit 钩子 — 提交前自动验证

📋 步骤 5.1：创建 git pre-commit 钩子 2 分钟
你说：
帮我创建 .git/hooks/pre-commit：

1. 提交前自动运行 python _test/run_verify.py
2. 验证不通过则 exit 1 阻止提交
3. 显示清晰的错误信息，提示可用 --no-verify 跳过
→ pre-commit 已创建 ✅
你说（测试钩子）：
故意加个语法错误，试试 git commit 会不会被阻止
→ [FAIL] 验证失败，提交已阻止！
→ 修复语法错误后再提交：通过 ✅
✅ 方法四完整：自动验证 + pre-commit 钩子，错误代码进不了仓库。

阶段六：MCP + Skills 配齐工具

📋 步骤 6.1：配置 MCP 服务器 5 分钟
你说：
在 settings.json 的 mcpServers 中添加两个 MCP 服务器：

1. filesystem：可访问项目目录和 D:/zdata/ 数据目录
2. sequential-thinking：用于复杂问题分析
→ MCP 已配置 ✅
→ Claude 现在可以直接操作文件系统、进行结构化推理
📋 步骤 6.2：创建 Skill 封装工作流 5 分钟
你说：
帮我创建一个 Skill，叫 data-check，功能是：

1. 输入：一个数据目录路径
2. 检查该目录下所有 CSV 文件的行数、最新日期
3. 输出：一个数据质量报告
4. 放在 .claude/skills/data-check/SKILL.md
→ Skill 已创建 ✅
→ 以后只需要说"检查一下 D:/zdata/dataAK中股/ 的数据质量"就能自动触发
✅ 方法三完整：MCP 扩展能力边界，Skills 固化工作流，Claude 不再只能聊天。

阶段七：提交存档 — 每功能一提交

📋 步骤 7.1：完成功能后提交代码 贯穿全程
你说（完成一个功能后）：
功能完成了，帮我：

1. 运行验证脚本确认没问题
2. 更新 CHANGELOG.md 或 README.md
3. git add + git commit，提交信息用 feat: 加了CSV读取功能
4. 不要 push
→ [PASS] 验证通过
→ README.md 已更新
→ Committed as "feat: 加了CSV读取功能" ✅
✅ 方法五完成：每功能一提交，随时可回滚，项目不失控。

完整实施流程（一键启动 Workflow 🚀）

如果你想一次搞定，直接用 Workflow 编排整个实施过程：

你说（终极版 — 一键实施 Harness）：
帮我启动一个 Harness 实施工作流，对当前项目做全套配置：

阶段一（并行初始化）：
agent 1：运行 /init 生成 CLAUDE.md，然后在末尾追加 AI 开发规则（工作流程、技术边界、代码规范、质量管理、版本控制）
agent 2：创建 .claude/settings.json，配置权限白名单（cd, python, pip, git, mkdir）和 hooks（自动 cd）
agent 3：创建 .gitignore，排除 __pycache__, *.log, .history/ 等

阶段二（并行工具链）：
agent 1：创建 _test/run_verify.py，做 Python 语法检查和文件完整性检查
agent 2：创建 .git/hooks/pre-commit，提交前自动跑验证
agent 3：在 settings.json 中配置 filesystem 和 sequential-thinking 两个 MCP 服务器

阶段三（汇总报告）：
汇总所有结果，生成一份 Harness 实施报告，列出已完成的配置项和路径
→ 三个阶段、最多 9 个 agent 并行工作
→ 几分钟内 Harness 全套配置完成 ✅
→ 生成报告：完成了 8 项配置
💡 实际项目注意：
  • 新项目 — 直接用一键 Workflow，10 分钟 Harness 全套到位
  • 已有项目 — 先让 Claude 扫描现有配置，再逐步添加缺失的部分
  • 大项目 — 先用 /plan 评估影响范围，再分阶段实施
  • 非 git 项目 — 跳过 pre-commit 钩子，先初始化 git 仓库
