第 10 课 · 用 Claude 实现 Harness 工程体系

前面 9 课讲了"怎么用好 Claude"，这一课讲"怎么管好 Claude"——Harness 工程体系。

为什么需要 Harness？

AI 编程最大的问题是不可控：

  • 你让它写一个功能，它可能把整个项目结构改了
  • 它用的技术栈可能和你的完全不搭
  • 写出来的代码看着能跑，实际全是 Bug
  • 改了三轮之后，回不到最初的版本了

Harness（缰绳）就是用传统软件工程的成熟经验，给 AI 套上缰绳——让它发挥创造力的同时，不至于失控。

✅ VSteach 实践： 本项目的 Harness 配置就是使用 Claude 本身来完成的—— 包括 CLAUDE.md 规则、settings.json 权限白名单、_test/run_verify.py 验证脚本、pre-commit 钩子，全部由 Claude 创建。

三层演进体系

层级说明效果
1. 提示词工程写好 Prompt，让 Claude 理解你要什么单次对话质量提升
2. 上下文工程配置 CLAUDE.md、项目知识，让 Claude 记住背景跨对话一致性好
3. Harness 工程权限、钩子、验证、MCP、Skills 全套体系项目级可控

五条核心方法（用 Claude 实现）

方法一：AGENTS.md 规则文件

目标：给 AI 划边界，从源头避免乱改技术选型、代码风格混乱。

用 Claude 实现：

  • 让 Claude 分析你的项目，生成一份 CLAUDE.md，包含项目背景、技术栈、代码规范
  • Claude 读一遍你的代码库，就能自动总结出合适的规则
  • 规则文件放在项目根目录，Claude 每次进入项目自动读取

效果：不用每次重复说明，Claude 自动知道"这个项目用什么 Python 版本、数据放哪里、不要改什么文件"。

方法二：先方案后编码

目标：减少 50% 以上返工，不做不需要的功能。

用 Claude 实现：

  • 新需求来了，先让 Claude 出方案："我想加个功能，先出个方案我看看"
  • Claude 给出要改哪些文件、怎么改、影响范围
  • 你过一遍，砍掉不需要的部分，确认后再让 Claude 动手
  • 或者使用 Plan 模式，Claude 先出规划，你确认后再进入编码

效果：避免 Claude 做了一堆你不需要的功能，或者方向错了改半天。

方法三：MCP + Skills 配齐工具

目标：让 Claude 能自主查资料、操作文件、调用外部工具。

用 Claude 实现：

  • MCP：在 settings.json 中配置 MCP 服务器（文件系统、搜索、数据库等），Claude 可以帮你配置
  • Skills：把常用工作流（如每日计划、代码审查）封装成 .claude/skills/ 下的 Skill 文件，Claude 帮你写这个文件

效果：Claude 不再局限于对话，能操作文件、查资料、运行工具。

方法四：自动测试验证

目标：避免"表面写完实际全是 Bug"。

用 Claude 实现：

  • 让 Claude 创建一个验证脚本（如 _test/run_verify.py），自动检查语法、文件完整性
  • 配置 Git pre-commit 钩子，每次提交前自动运行验证
  • Claude 每次改完代码，你只需要说"跑一下验证"，它就会自动运行

效果：语法错误、文件缺失在提交前就被发现，不会出现"写完跑不了"的情况。

方法五：文档 + Git 存档

目标：每功能一提交，随时可回滚。

用 Claude 实现：

  • 完成一个功能后，让 Claude 更新进度文档
  • 让 Claude 执行 git add + git commit，提交信息按规范格式（feat/fix/docs）
  • 所有临时文件放入 __temp/ 目录，不污染主目录

效果：任何时候出问题，都能回到上一个稳定版本，项目不会失控。

实际落地步骤（以 VSteach 为例）

这是我们在 VSteach 项目中实际执行的过程，可以作为参考模板：

步骤做什么让 Claude 做什么
1创建 settings.json"帮我创建项目的 .claude/settings.json，配置权限白名单和 hooks"
2强化 CLAUDE.md"在现有的 CLAUDE.md 末尾追加 AI 开发规则章节"
3创建验证脚本"创建 _test/run_verify.py，检查 Python 语法和文件完整性"
4配置 MCP"在 settings.json 中添加 filesystem 和 sequential-thinking MCP 服务器"
5创建 Skills"把 xxx 工作流封装成 Skill"
6pre-commit 钩子"创建 .git/hooks/pre-commit，提交前自动跑验证"
7完善 .gitignore"帮我完善 .gitignore，排除日志和临时文件"
💡 关键原则：
  1. 从易到难 — 先做 settings.json + CLAUDE.md（10 分钟），再做验证脚本（15 分钟），再做 MCP 和 Skills（按需）
  2. 触发条件 — 不提前做还没用到的东西，等需要的时候再配置
  3. 让 Claude 做 — 这些配置大部分都可以让 Claude 自己创建，你只需要说"帮我做"
  4. 先方案后编码 — 让 Claude 出方案，你确认，再执行——这条本身就是 Harness 的核心

效果对比

场景没有 Harness有 Harness
新需求直接写代码，改完发现方向错了先出方案，你确认，再编码
改代码Claude 可能改了不该改的文件CLAUDE.md 约束了边界，不会越界
提交代码语法错误被提交到仓库pre-commit 钩子阻止了错误提交
查资料Claude 用训练数据里的旧知识MCP 联网搜索，拿到最新信息
重复工作每次都要重新描述规则Skill 封装好，一句话调用
出问题回退困难，不知道改了哪里每功能一提交，随时回滚
🎯 总结： Harness 不是一次性的配置工作，而是一种工作习惯。 每次开发前想一下"我有没有套好缰绳"——时间长了就成了肌肉记忆。
